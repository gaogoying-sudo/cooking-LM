Overview

Cooking LM

Cooking LM（Cooking Language / Intelligence Model 工作台）

面向菜谱理解、专家经验沉淀、策略决策与设备交付的统一智能能力中枢

Cooking LM 不是单一模型，也不是简单的菜谱问答工具。它围绕“客户需求理解、菜品匹配与生成、工艺和参数适配、设备执行交付、结果反馈优化”构建系统性能力底座，真正沉淀的是菜谱知识、专家经验、调整逻辑和交付策略，逐步形成公司自己的智能资产与交付能力。

当前阶段聚焦

  • 先沉淀知识主干与规则主干
  • 先让系统承接口味工程师的部分现场工作

Boundary Definition

Cooking LM 不是什么

这一块用于帮助团队快速收敛边界。Cooking LM 的建设重点是知识、策略与交付能力沉淀，而不是做一个看起来像 AI 的表层入口。

不是单纯的聊天机器人

它不以自然语言交互本身为目标，而以可持续交付能力建设为目标。

不是简单的菜谱搜索器

它不只是返回菜谱，而是要理解菜品、工艺、参数和设备适配关系。

不是临时把数据交给大模型求答案

模型只负责局部推理与加工，核心资产必须沉淀在自有知识和规则体系中。

不是语音入口型展示 MVP

页面或入口只是承载层，真正需要建设的是数据主干、知识主干和交付闭环。

North Star

终极目标

北极星不是做一个会说话的模型，而是让系统逐步承接原本依赖口味工程师现场完成的迁移、调试和交付工作。

已有菜迁移交付

帮助客户把原有菜品从人工大锅经验迁移到设备侧执行。

专家经验显性化

把口味工程师在现场的判断、调整和教学经验沉淀下来。

菜谱与参数结构化

把菜谱、步骤、食材、调味料和设备参数变成结构化对象。

北极星目标 让客户买到设备后，逐步减少对口味工程师现场支持的依赖

通过沉淀菜谱知识、专家经验、参数调整逻辑和交付策略，逐步支撑已有菜迁移、录菜、调试与交付，不再每次都依赖工程师到现场兜底。

策略与规则协同

明确哪些场景可自动建议，哪些必须受规则和人工确认约束。

客户自助录菜辅助

逐步降低客户使用设备和完成录菜、调试、交付的门槛。

反馈驱动迭代

通过结果表现、使用反馈和调整记录不断修正知识与策略。

Architecture

总体系统架构

这五层不是纯技术拆分，而是业务能力如何从原始资产一路沉淀为可交付系统的结构表达。

01

原始数据层

沉淀原始菜谱、录菜记录、设备日志和多模态交付素材。

菜谱数据 录菜过程 设备日志 客户需求 专家记录 图像/视频/语音素材
02

结构化知识层

把原始数据提炼为结构化知识对象，是长期资产底座。

Dish Specification Recipe Intent 工艺阶段 参数对象 调整动作 调整原因 标杆菜谱 经验规则
03

策略规则层

决定系统能不能做、做到什么程度，以及何时需要人工兜底。

推荐策略 适配规则 风险阈值 置信度控制 人工确认机制 设备约束规则
04

模型推理层

模型只负责局部推理、抽取、匹配和候选生成，不替代知识与规则主干。

需求解析 相似检索 方案对比 参数建议 多模态理解 解释生成
05

执行交付层

把建议输出到客户与设备端，并将结果重新回流系统。

客户交互 录菜辅助 参数建议输出 设备执行支持 交付确认 反馈回流
Core Assets

公司真正的资产

外部模型可以参与推理和加工，但公司的长期护城河，在于把菜谱、经验、策略、规则和设备交付能力沉淀为自己的知识与执行系统。

核心资产不是外部大模型本身，而是可持续沉淀、可持续复用、可持续放大的业务知识与交付能力。
菜谱数据 专家经验 调整逻辑 交付策略 设备适配知识 结果反馈数据 多模态素材 结构化知识与规则体系
AI Role Split

AI 能力参与方式

这里不是 agent 角色划分，而是 AI 在 Cooking LM 中的三种典型参与形态：离线加工、在线推理、逐步固化。

Offline

离线加工

把历史资产变成结构化知识候选，结果必须回写到公司系统。

  • 历史菜谱解析
  • 专家纪要抽取
  • 同菜多版本差异分析
  • 图像 / 视频 / 语音素材理解
  • 经验规则候选整理
  • 数据清洗与标准化
→
Online

在线推理

基于当前少量上下文完成局部认知任务，由系统先检索、筛选、组织上下文。

  • 客户需求解析
  • 相似菜谱候选理解
  • 当前场景方案对比
  • 参数建议生成
  • 结果解释与说明
  • 多模态补充理解
→
Solidify

能力固化

把高频、稳定、可验证能力沉淀为规则、小模型或本地逻辑。

  • 单位换算
  • 参数边界检查
  • 设备兼容判断
  • 已验证映射规则
  • 基础排序 / 打分
  • 固定约束判断
Context Strategy

为什么不能每次把全量数据上传给大模型

真正可持续的系统不是“每次临时喂全量数据再求答案”，而是先完成知识沉淀，再把精简上下文交给模型处理当前任务。

错误路径

让模型反复重新理解所有历史数据

  • 每次请求都依赖重新上传大量数据
  • 成本高，上下文容易混乱
  • 响应速度慢
  • 知识无法沉淀
  • 规则难稳定、结果难复盘
  • 公司资产长期停留在 prompt 里
正确路径

公司系统先沉淀，再把少量相关上下文交给模型

  • 原始数据先进入公司系统
  • 系统先做结构化和知识沉淀
  • 先检索命中少量相关上下文
  • 再把精简上下文交给模型
  • 模型只做当前任务推理
  • 结果回写知识与流程体系

System Core

系统内必须沉淀的能力

公司的长期护城河，不在于“是否拥有一个自建大模型”，而在于是否把菜谱、经验、策略、规则、参数逻辑和设备适配能力沉淀成自己的知识与执行系统。

这些能力属于底座、主干、不可外包部分，必须在公司自己的系统中可控、可复盘、可版本化。
数据存储与版本管理 菜谱主数据 结构化知识对象 调整动作与原因记录 标杆菜谱库 规则体系 风险阈值体系 检索索引 人工确认流程 反馈回流机制
Deployment Decision

本地化与模型部署判断

当前真正的短板不在“有没有自己的大模型”，而在于数据、知识、规则和流程主干是否已经建立起来。

判断一

当前阶段：不建议先投入重资产模型部署

  • 知识尚未结构化
  • 高频 AI 任务尚未稳定
  • 真正短板不是模型，而是数据与知识工程
  • 缺少维护私有模型服务的成本与能力
  • 现在更值钱的是规则、知识和流程沉淀
↓
判断二

出现这些条件时，再考虑本地化 / 私有化

  • 高并发固定任务已经形成
  • 云端成本持续升高
  • 延迟要求明显提升
  • 隐私隔离诉求明显提高
  • 已经沉淀出可蒸馏的小任务
  • 团队具备维护推理服务能力
↓
判断三

真要投入算力时，优先用于这些方向

  • 小模型推理
  • 本地 OCR / ASR / 多模态预处理
  • embedding / rerank 替代
  • 固定分类任务
  • 高频参数判断任务
  • 局部专用模型服务

不是一上来追求自建 GPT 替代品，而是优先固化高频、稳定、成本敏感的局部任务。

Roadmap

建设路线

当前阶段不是急着做“大而全智能体”，而是先建立资产、知识、策略与交付协同的主干能力，再逐步让 AI 发挥价值。

Phase 1

资产沉淀与知识主干建立

先把原始数据、知识对象、规则边界和版本体系建立起来。

  • 数据对象统一
  • 知识结构建模
  • 规则边界定义
  • 核心流程梳理

进入下一阶段：统一基础数据对象，形成首批知识结构与标杆菜谱，并跑通一条 AI 加工链。

投入重点：数据治理、知识结构设计、规则边界梳理。暂不投入重资产 GPU 和自建生成模型。

Phase 2

AI加工与检索增强

让 AI 成为“加工厂”和“检索辅助器”，提升历史资产可用性。

  • 历史数据批量加工
  • 结构化抽取与归类
  • 相似检索与召回增强
  • 候选知识生成

进入下一阶段：检索与召回可用，已形成基础规则边界，并识别高频交付场景。

投入重点：AI 批处理、检索增强、结构化抽取。开始探索小规模模型服务。

Phase 3

策略协同与交付辅助

把检索、规则、建议和人工确认组合成稳定交付链。

  • 建议生成与解释
  • 风险控制与阈值判断
  • 人工确认与回退机制
  • 客户录菜辅助

进入下一阶段：建议质量稳定，已形成反馈回流数据，并识别可固化高频能力。

投入重点：建议生成、风险控制、客户辅助交付。开始评估是否需要小模型或本地推理。

Phase 4

反馈闭环与持续优化

通过执行结果和反馈数据持续修正知识、规则、建议质量和能力边界。

  • 反馈回流
  • 规则修正
  • 知识再沉淀
  • 高价值能力逐步固化

可能开启：OCR / ASR / rerank / embedding 替代，以及局部专用模型服务。

重投入判断：仅当高频任务稳定、成本和延迟问题显著、且团队具备维护能力后，再评估局部模型与算力投入。

Priority Scope

当前重点

当前阶段的关键，不是把所有能力一次性做全，而是先抓最贴近真实交付价值、最能沉淀公司资产的能力主线。

当前先做

  • 已有菜迁移交付场景
  • 标杆菜谱沉淀
  • 专家经验显性化
  • 客户自助录菜辅助
  • 数据与知识主干建设
阶段目标 / 达成标准

形成统一知识对象，建立首批标杆菜谱，完成至少一个高频交付场景闭环，并验证基础 AI 加工能力可用。

暂不优先

  • 开放式创意菜自动生成
  • 复杂个性化风味创作
  • 全量设备场景一次性覆盖
  • 重度可视化分析看板
延后原因 / 未来开启条件

等主链路稳定、数据输入质量达标、检索与规则层可用，并且出现明确业务场景拉动后再逐步打开。

当前不做

  • 自建通用大模型
  • 大而全私有化替代 GPT
  • 重资产 GPU 集群先行
  • 脱离数据治理直接做端到端 AI
当前不做原因 / 重投入触发条件

只有当高频任务稳定、云成本明显上升、延迟与隐私要求增强，且团队具备维护局部模型能力时，才考虑重投入。

Strategy Summary

当前技术策略总结

先把公司的数据、知识、规则和流程沉淀下来，让大模型参与局部加工和推理；等高频任务稳定后，再逐步把可固化能力沉淀为规则、小模型或本地执行能力。
