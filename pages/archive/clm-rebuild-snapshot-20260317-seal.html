项目定义

Project Goal

第一阶段要替代的是口味工程师的首轮到场交付工作

在口味工程师不到客户现场的前提下，使客户能够通过语音、文字、图片和设备交互表达需求，并由系统在设备侧生成一版高可用的初始菜谱方案，目标结果接近人工交付水平，并作为后续微调和收敛基础。

建设对象 不是单独模型，也不是单独 Agent，而是围绕需求输入、知识、规则、模型和输出执行组织起来的交付系统。
阶段要求 先把客户表达需求、系统生成高可用初始菜谱、人工继续微调收敛这一条主链跑通。
交付标准 目标不是一次满分，而是让系统能替代首轮到场交付，把初版质量拉到可试做、可继续收敛的水平。

Layer 01

原始数据接入层

把可能有价值的原始素材和原始证据，以可追踪、可登记、可管理的方式接入系统，形成原材料池。

输入数据库、菜谱记录、设备日志、交付纪要、图片、视频、音频、专家经验。
处理动作数据接入、来源登记、元数据标记、时间戳记录、权限分类、原始证据归档。
输出原始素材池、原始证据池、已登记数据源、待解析素材队列。
当前建设重点先接稳定、高频、与样板任务直接相关的素材，不做无边界抓取。
Layer 02

结构化加工层

把不同来源、不同模态、不同粒度的原始素材映射到统一对象体系里的字段、标签、关系、时间线和证据链接。

输入原始素材池、原始证据池、已登记数据源。
处理动作文本抽取、音频转写、视频切片、步骤识别、参数识别、时间线抽取、对象映射、证据挂载。
输出结构化记录、对象挂载结果、标签关系、待沉淀候选。
当前建设重点先围绕菜品、菜谱版本、工艺步骤、参数项、调整动作和调整原因这些核心对象打底。
Layer 03

知识沉淀层

不是把所有结构化数据自动升华成知识，而是从中筛选、归纳、验证出可复用、可解释、可决策的经验、规则、模式和模板。

输入结构化记录、相似记录聚合结果、人工反馈、历史交付结果。
处理动作规律归纳、候选规则生成、经验条目提炼、失败模式聚类、风险映射、验证状态管理。
输出经验规则、放量规则、调参原因模板、失败模式、风险策略、标杆菜谱。
当前建设重点不做全量世界建模，先围绕任务闭环沉淀能直接支撑决策的知识类型。
Layer 04

检索层

检索层负责找。

输入用户需求、结构化任务、当前上下文、查询条件。
处理动作数据库检索、知识库命中、向量相似召回、候选集合组织。
输出候选菜谱集合、候选案例集合、候选规则集合、命中来源。
当前建设重点先保证相关上下文能找全，尤其支撑样板任务和场景 1 的强相关召回。
Layer 05

规则与策略层

规则与策略层负责裁。

输入检索结果、设备上下文、用户角色、知识状态、风险等级、任务类型。
处理动作过滤、约束、边界控制、审核要求判断、流程收口、自动下发限制。
输出可进入下一步的候选范围、过滤原因、当前可输出建议等级、是否需要人工审核。
当前建设重点先把设备约束、知识状态、风险阈值和人工确认条件做成稳定底线。
Layer 06

模型推理层

模型推理层负责想。

输入规则筛后的少量候选、当前需求描述、任务目标、必要业务约束。
处理动作语义理解、比较、排序、解释、建议生成、缺口识别。
输出排序后的建议结果、推荐理由、参数建议、风险说明、缺失信息提示。
当前建设重点让模型只处理局部认知任务，不让它承担记忆、规则和执行主逻辑。
Layer 07

输出执行层

把建议结果转成真实业务动作。

输入建议结果、风险等级、审核结论、任务状态。
处理动作输出设备侧菜谱 / 工艺包 / 参数包，触发补问、审核、转人工与结果回写。
输出设备执行结果、内部辅助交付结果、回写记录、知识建设任务。
当前建设重点把“系统结果如何真正落到交付工作里”先做扎实，不让系统停在建议层。
Delivery Result

最终交付的是什么

先从业务可见结果倒推，而不是一上来罗列数据库、Agent 和向量库。

设备侧可执行菜谱 / 工艺包 / 参数包

这是客户侧和设备侧真正接收到的可执行结果，不是口头建议。

内部知识与决策系统

这是持续积累规则、案例、风险边界和决策能力的主干系统。

交付辅助工作台

承接补问、审核、解释、转人工和结果回写，而不是只有一个对话框入口。

Core Objects

核心对象

对象层要稳，后面的知识和决策才能稳。

菜品

一句话定义：承接这道菜的业务身份与风味定位。

关键字段示例：dish_id、菜品名、风味标签、目标场景。

与谁关联：菜谱版本、设备上下文、评价结果。

当前建设状态：对象骨架明确，字段细化中。

为什么重要：它是所有版本、规则和案例的主对象。

菜谱版本

一句话定义：承接某一道菜在某个阶段、某种设备或某次调优下的具体版本。

关键字段示例：version_id、参数包、适配设备版本、状态。

与谁关联：菜品、工艺步骤、参数项、评价结果。

当前建设状态：主字段明确，版本链规则待细化。

为什么重要：它是交付、复刻和回写的直接载体。

工艺步骤

一句话定义：承接菜谱执行中的动作顺序和阶段变化。

关键字段示例：step_id、动作、阶段、时长、触发条件。

与谁关联：菜谱版本、参数项、调整动作。

当前建设状态：步骤和阶段概念已锁，执行映射待补。

为什么重要：没有步骤对象，就无法把菜谱真正转成可执行流程。

参数项

一句话定义：承接设备侧可以真正控制和调整的变量。

关键字段示例：火力、时长、速度、投料量、阈值。

与谁关联：工艺步骤、设备上下文、评价结果。

当前建设状态：基础参数清单明确，依赖关系待沉淀。

为什么重要：这是系统输出到设备侧时最关键的执行对象。

调整动作 / 调整原因

一句话定义：把“改了什么”和“为什么改”绑定成一组可学习的记录。

关键字段示例：调整动作、调整前后差异、触发原因、人工备注。

与谁关联：菜谱版本、参数项、评价结果。

当前建设状态：结构框架明确，经验公式仍待补。

为什么重要：这是专家经验显性化和后续规则沉淀的关键桥梁。

设备上下文 / 评价结果

一句话定义：把建议始终放回具体设备和试做结果里理解。

关键字段示例：设备型号、执行状态、评分、缺陷项、反馈标签。

与谁关联：菜谱版本、参数项、调整动作。

当前建设状态：设备维度已有基础，评价结构仍在细化。

为什么重要：它保证系统不是抽象建议，而是贴着真实执行效果生长。

Knowledge State

知识状态

AI 生成候选，不等于直接成为有效知识。

草稿

状态定义：原始整理结果和初始提炼内容。谁生成：AI 或人工初步整理。谁审核：暂不进入正式审核。典型内容形态：字段草稿、候选标签、原始摘录。是否可参与线上决策：不可以。是否可进入规则 / 检索 / 建议链路：不可以。如何进入下一状态：经人工筛选后升为候选。常见问题：噪声多、结构不稳。

↓
候选

状态定义：AI 或人工归纳出的候选知识。谁生成：结构化加工结果归纳、人工总结、模型候选生成。谁审核：业务或专家做初筛。典型内容形态：候选规则、经验条目、失败模式草稿。是否可参与线上决策：原则上不直接参与。是否可进入规则 / 检索 / 建议链路：可进入离线整理，不进入线上强决策。如何进入下一状态：进入待验证。常见问题：看起来合理但证据链不足。

↓
待验证

状态定义：进入人工验证、样板任务验证或流程验证。谁生成：候选知识经过初筛后转入。谁审核：口味工程师、业务负责人或样板任务验证流程。典型内容形态：待确认规则、待验证放量关系、待验证风险条目。是否可参与线上决策：仅可作为弱参考。是否可进入规则 / 检索 / 建议链路：可进入低权重检索参考，不进入强规则链。如何进入下一状态：验证通过后升为已验证。常见问题：场景适用范围不清。

↓
已验证

状态定义：已经过验证，可以进入线上链路。谁生成：验证流程确认后的稳定知识。谁审核：最终由业务或专家确认。典型内容形态：标杆菜谱、已验证规则、稳定风险策略。是否可参与线上决策：可以。是否可进入规则 / 检索 / 建议链路：可以进入检索、规则和建议主链。如何进入下一状态：被新版本替代或不适用时转失效。常见问题：需要持续监控是否过时。

↓
失效

状态定义：不再适配当前边界或被新版本替换。谁生成：新验证结果、业务变化或设备变化触发。谁审核：人工确认失效范围。典型内容形态：旧规则、旧版本模板、过期风险映射。是否可参与线上决策：不可以。是否可进入规则 / 检索 / 建议链路：只保留历史参考，不进入当前线上主链。如何进入下一状态：作为历史引用保留。常见问题：历史依赖未及时解除。

这里不再泛讲原则，而是直接围绕“我要一个北京口味的宫保鸡丁”把从输入到输出、再到失败降级的整条链讲清楚。

1. 输入是什么

用户原始输入文本是“我要一个北京口味的宫保鸡丁”。当前模糊点包括：偏甜还是偏酸、偏辣程度、试菜份量、设备型号、客户最终是要一版参考还是要接近可直接执行的初版方案。当前目标是把模糊输入先转成可检索、可缩圈的任务表达。

↓
2. 原始数据接入层会找什么

系统会先拉内部已有的宫保鸡丁相关菜谱、北方 / 北京风味相关历史记录、相似客户案例、外部参考素材以及口味工程师经验条目。当前输入是客户需求和设备侧上下文，当前动作是把相关原始素材接进来并登记来源，当前输出是一个可被继续处理的原始证据池。当前风险是：如果这一步命中的原始素材太弱，后面所有层都会被迫降级。

↓
3. 结构化加工层会抽什么

系统要把原始素材里的菜品对象、风味标签、关键食材、关键工艺、设备上下文和风险标签抽出来，并挂到统一对象体系上。当前输入是原始证据池，当前动作是做文本抽取、标签识别、步骤与参数抽取、证据挂载，当前输出是宫保鸡丁相关的结构化对象集合。当前风险是：风味标签和调整原因如果抽不稳，后续知识命中会偏弱。相关底层定义来自“阶段性工作产出 → 菜谱维度展开”。

↓
4. 知识沉淀层命中什么

在这个案例里，知识沉淀层真正重要的是已验证规则、失败模式、北京 / 北方风味偏好经验、相似菜映射知识和参数依赖知识。当前输入是结构化对象集合，当前动作是命中已有知识条目并判断哪些是已验证、哪些还只是候选，当前输出是可进入检索与规则层的知识资产包。当前风险是：如果只有候选知识而没有已验证知识，输出等级必须下降。命中的知识条目来源可回看“阶段性工作产出 → 经验规则 / 经验公式”。

↓
5. 检索层在这个案例里“找”的是什么

检索层负责找相关上下文。它找的是候选菜谱、候选案例、候选规则，而不是直接给答案。当前输入是任务表达和知识资产包，当前目标是尽量找全跟“宫保鸡丁 + 北京口味 + 当前设备”相关的上下文，当前动作是数据库检索、知识库命中和相似召回，当前输出是候选菜谱范围、命中案例和命中规则集合。当前限制是：检索层只负责找，不负责判断这些候选能不能直接进入输出。

↓
6. 规则与策略层在这个案例里“裁”的是什么

规则层裁的是边界、风险和流程。它会去掉未验证、失效、不适配当前设备或与当前任务目标不匹配的候选；它还要判断当前是否必须补问“偏甜 / 偏酸 / 偏辣”，是否必须人工审核，是否允许进入强建议模式。当前输入是候选菜谱、候选案例、候选规则和设备上下文，当前动作是过滤与收口，当前输出是一个被压缩后的可用候选范围和当前允许输出的建议等级。当前限制是：规则层不负责给推荐理由，但会决定模型最多能往下做多深。

↓
7. 模型推理层在这个案例里“想”的是什么

模型层想的是：这里的“北京口味”更接近哪类风味路径，候选版本之间哪个更贴近当前场景，是否需要继续补问“偏甜 / 偏酸 / 偏辣”，以及如何形成推荐理由。当前输入是规则筛后的少量候选和必要业务约束，当前目标是进一步缩圈而不是擅自发明知识，当前动作是理解、比较、排序、解释和建议生成，当前输出是推荐理由、候选排序和参数建议等级。当前风险是：如果候选本身弱，模型层只能做弱建议与缺口识别，不能装作很确定。

↓
8. 最终输出是什么

在数据充分时，最终输出可以是 2–3 个候选菜谱范围、推荐理由、参数建议和风险提示；在数据一般时，最终输出是建议级候选 + 补问；在数据零散时，最终输出是相似案例参考和缺失项；在数据缺失时，最终输出只能是下一步引导。当前输出的核心不是“给没给答案”，而是“给到了什么等级的结果”。

↓
9. 数据不足时怎么降级

当数据不足或知识不足时，系统要按顺序降级：先补问，再给弱建议，再转人工，最后进入知识建设队列。当前动作不是硬撑出一个假答案，而是明确告诉系统和人下一步该怎么推进。这也是为什么数据少不等于系统停机，输入模糊不等于系统不能工作。

Risks and Dependencies

风险与依赖

把依赖说清楚，计划才不是聊天式路线图。

关键依赖

对象模型不锁，结构化加工和知识沉淀无法稳定推进。

主要风险

如果场景边界重新放宽，样板任务就会失焦，导致整个计划拖散。

控制方式

先守住场景 1 主线和样板闭环，再逐步打开更多能力。
