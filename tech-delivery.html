Definition

Cooking LM 技术交付工作台

从语料加工、知识沉淀、模型推理到 Agent 编排与输出执行的技术系统设计图

最终交付物不是单独一个模型，而是一套以公司私有语料为原材料、通过结构化抽取、知识沉淀、检索、规则和模型推理组成的内部智能系统。

交付目标

把语料加工成可追踪、可审核、可执行的内部智能系统。

能力边界

模型负责理解与推理，知识、规则与状态必须沉淀在公司系统中。

最终输出

输出给工作台、审核链路与设备执行结果，而不是一个孤立模型答案。

Layer Map

技术分层与处理链路

原始数据接入层

已完成

做什么：接入文本、素材、日志和记录。输入：语料。输出：可追踪原始数据。

结构化加工层

进行中

做什么：抽取字段与对象。输入：原始语料。输出：结构化记录与候选对象。

知识沉淀层

进行中

做什么：沉淀知识资产。输入：结构化记录。输出：规则、标杆菜谱、知识对象。

检索层

进行中

做什么：召回相关上下文。输入：查询与知识资产。输出：候选内容集合。

规则与策略层

进行中

做什么：判断边界和风险。输入：候选内容。输出：可用建议范围与审核要求。

模型推理层

进行中

做什么：理解、解释和候选生成。输入：精简上下文。输出：建议与解释结果。

Agent 工作流层

待建设

做什么：把检索、规则、模型、审核串成任务链。输入：任务请求。输出：完整工作流执行结果。

待建设

Module Map

完整交付需要建设的技术模块

每个模块都对应清晰职责、交付项和核心对象，避免系统建设停留在抽象概念层。

数据底座开发

职责：建立统一存储和版本管理。

交付项：数据 schema、版本策略。

对象：dataset_id、version_id。

数据接入与预处理

职责：接入语料并做标准化。

交付项：接入管道、清洗规则。

对象：source_type、raw_record。

AI 结构化抽取服务

职责：把语料转成字段与对象。

交付项：抽取服务、候选对象。

对象：field_map、extraction_job。

知识沉淀与审核模块

职责：候选知识审核入库。

交付项：状态流转、审核记录。

对象：knowledge_id、review_log。

向量检索模块

职责：召回相似内容与上下文。

交付项：索引、召回接口。

对象：embedding_id、recall_set。

规则引擎与策略配置

职责：控制约束、边界和审核。

交付项：策略配置、阈值规则。

对象：rule_id、risk_level。

在线决策服务

职责：聚合检索、规则和模型。

交付项：建议接口、解释接口。

对象：decision_context、response_pack。

Agent 编排层

职责：串联多个能力完成任务。

交付项：流程模版、任务编排。

对象：workflow_id、task_state。

前端 / 工作台

职责：承载审核、查询和交付。

交付项：操作界面、状态展示。

对象：workspace_state、review_action。

评测与监控

职责：跟踪质量、性能和风险。

交付项：评测集、监控面板。

对象：metric_id、evaluation_result。

Knowledge Lifecycle

知识状态流转与审核机制

AI 只能生成候选，不能直接成为有效知识；知识必须经过状态流转、审核、生效和失效管理。

草稿

谁生成：人工录入或初始整理。

谁审核：暂无，先做内容整理。

如何留痕：保留来源、录入人、时间。

进入下一状态：完成基础字段整理后进入候选。

↓
候选

谁生成：AI 抽取或批处理加工。

谁审核：审核工作台初筛。

如何留痕：保留抽取任务、模型版本、原始证据。

进入下一状态：对象绑定完整后进入待验证。

↓
待验证

谁生成：系统自动聚合候选对象。

谁审核：领域专家、口味工程师。

如何留痕：记录审核意见、版本与修改原因。

进入下一状态：通过验证后进入已验证。

↓
已验证

谁生成：验证通过后的正式知识条目。

谁审核：最终由审核链路确认生效。

如何留痕：保留版本、生效时间、责任人。

进入下一状态：规则变化或效果退化时转入失效。

↓
失效

谁生成：系统或人工标记失效。

谁审核：规则负责人或专家确认。

如何留痕：保留失效原因、时间和替代版本。

进入下一状态：如重新修订，可回到草稿重新加工。

Investment Gate

什么时候开始考虑模型、硬件和本地化重投入

当前阶段不建议优先投入自建生成模型；只有当高频任务稳定、成本和延迟问题显著、且团队具备维护能力后，才逐步评估局部模型和算力投入。

判断一

开始投入生成模型相关学习 / 评估

  • 已沉淀稳定高频任务
  • 已明确值得固化的任务
  • 已有可用于蒸馏 / 训练的样本
↓
判断二

开始投入本地推理或专用模型

  • 延迟要求显著提升
  • 高频任务稳定
  • 云成本明显上升
  • 本地化收益明确
↓
判断三

开始投入硬件 / 显卡

  • 已过概念验证阶段
  • 有固定高频推理任务
  • 团队具备维护能力
  • ROI 可以成立

原则：不是先造大模型，而是先固化高频稳定局部能力。
